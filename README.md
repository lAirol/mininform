# Регистрация СМИ: технический гид для разработчиков



## 1) Что это за проект

Проект собирает большую многошаговую форму, валидирует данные на клиенте и на сервере, формирует PDF заявления и сохраняет артефакты в `uploads/files/form`. `1.html` - это view

Основные файлы:

- `1.html` — структура формы (шаги, статические поля, контейнеры для динамики).
- `main.js` — клиентская логика: шаги, валидация, сборка JSON, отправка, обработка ответов сервера, UI состояния.
- `add_forms.js` — динамические блоки (учредители, собственники, спонсоры, вложенные структуры).
- `validate_smi_json.php` — серверная валидация JSON и формирование человекочитаемых ошибок.
- `generate_pdf.php` — генерация PDF + HTTP API endpoint для POST JSON.

## 2) Поток данных end-to-end

1. Пользователь заполняет форму в `1.html`.
2. `main.js` валидирует текущий шаг и дополнительные бизнес-правила по шагам.
3. `main.js` собирает JSON по `data-path` через `buildJsonFromForm()`.
4. JSON отправляется `fetch` на `generate_pdf.php`.
5. `generate_pdf.php`:
   - парсит входной JSON,
   - запускает `validateSmiJson(...)`,
   - при успехе генерирует PDF через `SmiPdfGenerator`,
   - возвращает JSON ответ `message`/`errors`.
6. `main.js`:
   - при успехе показывает экран «Заявление отправлено»,
   - при `400` маппит ошибки полей в UI и фокусирует первое проблемное поле,
   - при `500` показывает toast с общей ошибкой сервера.

## 3) Главная идея разметки: `data-path`

`data-path` — это контракт между HTML/JS/PHP.  
По нему поле попадает в JSON и по нему же сервер возвращает адрес ошибки.

Примеры:

- `data-path="mainInfo.name"` -> `{ "mainInfo": { "name": "..." } }`
- `data-path="office.chiefEditor.name"` -> вложенный объект
- `data-path="founders.0.name"` -> массив с индексами

Важно:

- Поля без `data-path` в JSON не попадают.
- Если значение пустое (`''`), `setPath(...)` его пропускает.
- Для `checkbox`/`radio` учитываются только `checked`.
- Для `select[multiple]` каждое выбранное значение добавляется в массив.

## 4) Ключевые `data-*` приёмы в проекте

### `data-path`

Источник истины для:

- сборки JSON (`buildJsonFromForm`),
- серверной валидации,
- отображения серверных ошибок в поле.

### `data-validate`

Клиентские валидаторы:

- `email`
- `phone`
- `phoneCode`
- `percent`

Используется в `validateField(...)`.

Подробно по каждому:

- `data-validate="email"`  
  Регулярка: `^[^\s@]+@[^\s@]+\.[^\s@]+$`.  
  Ошибка: «Укажите корректный адрес электронной почты».

- `data-validate="phone"`  
  Разрешённые символы: цифры, `+`, `(`, `)`, `-`, пробел; минимум 5 символов.  
  Регулярка: `^[0-9+()\-\s]{5,}$`.  
  Ошибка: «Укажите корректный номер телефона».

- `data-validate="phoneCode"`  
  Только 2-6 цифр.  
  Регулярка: `^\d{2,6}$`.  
  Ошибка: «Укажите корректный код города (2–6 цифр)».

- `data-validate="percent"`  
  Число в диапазоне `0..max`, где `max` берётся из `data-max-percent`, иначе `100`.  
  Поддерживаются `,` и `.` как разделители ввода (внутри проверки приводится к `.`).  
  Ошибка: «Укажите значение от 0 до 100» или «Для этого поля диапазон значений ...».

Полезная деталь: часть `data-validate` выставляется автоматически в `main.js` по суффиксу `data-path`:

- `...email` -> `email`
- `...phoneCode` -> `phoneCode`
- `...phone` -> `phone` (кроме `phoneCode`)
- `...percent` / `...bynVolume` -> `percent`

### `data-required="true"`

Форсирует required-поведение там, где нельзя/неудобно ставить стандартный `required` напрямую (например, динамические карточки из шаблонов).

### `data-ignore-container`

Ключевой механизм «этот блок временно выключен»:

- при наличии атрибута поля внутри не валидируются,
- эти поля не участвуют в сборке JSON.

Используется для переключаемых веток формы (`Учредитель` vs `Владелец сетевого издания`, вложенные блоки при госоргане и т.д.).

### Специализированные управляющие `data-*` из `add_forms.js`

- `data-add-physical-founder`, `data-add-jur-founder` — добавление карточек учредителей.
- `data-add-office-owner` — добавление собственников.
- `data-add-sponsor` — добавление источников финансирования.
- `data-state="editing|saved"` — состояние карточки.
- `data-is-new="1|0"` — новая/существующая карточка.
- `data-parent-path`, `data-owner-path`, `data-founder-path` — навигация по вложенным структурам.
- `data-sum` — вычисленная сумма долей.

## 5) Как работает `main.js`

## 5.1 Шаги формы и навигация

- Форма разбита на `.step`.
- `showStep(...)` переключает шаг, обновляет прогресс и текст кнопки.
- Перед переходом вперёд вызывается `validateStep(current)`.
- На последнем шаге вызывается `sendData()`.

## 5.2 Базовая валидация полей

`validateField(...)` проверяет:

- обязательность (`required`/`data-required`),
- email/телефон/код телефона/проценты,
- числовые поля.

Отдельно учтён `select[multiple]`:  
выбранным считается только осмысленный option (не `disabled`, не пустой `value`).

Мелочи, которые важно знать:

- `required` на отключённом поле (`disabled`) не срабатывает, потому что такие поля обычно находятся в ветках с `data-ignore-container`.
- Для динамических карточек лучше использовать и `required`, и `data-required="true"` — это защищает от edge-case при временных преобразованиях DOM.
- Для процентов используйте `data-validate="percent"` и при необходимости `data-max-percent="20"` (или другое значение).
- Для телефонных и числовых полей проект дополнительно чистит ввод на событии `input` и блокирует лишние символы на `keydown`.

## 5.3 Дополнительная бизнес-валидация по шагам

В `additionalStepValidate(...)`:

- Шаг 2: должен быть заполнен либо блок учредителей, либо владелец сетевого издания.
- Шаг 3: помещение, собственники, квалификация главреда.
- Шаг 4: специализация обязательна «любым способом»:
  - либо выбран список `mainInfo.specialization`,
  - либо включено `specializationOtherEnabled` и заполнено `specializationOther`.
- Шаг 5: финансирование + хотя бы один источник.
- Шаг 6: чек подтверждения.

## 5.4 Сборка JSON

`buildJsonFromForm(container)`:

- берёт все `[data-path]`,
- исключает всё внутри `data-ignore-container`,
- раскладывает значения по пути через `setPath(...)`,
- объединяет `founders` и `founders_phys` в один массив `founders` (`mergeFounders`).

Это критично: сервер валидирует `founders[...]`, а UI может генерировать обе ветки (`founders` и `founders_phys`).

## 5.5 Отправка и обработка ответа сервера

`sendData()`:

- валидирует все шаги (`_validateAllSteps`),
- отправляет POST JSON на `generate_pdf.php` (либо любой другой указанный в нокстанте `send_url`),
- читает ответ и пытается распарсить JSON независимо от заголовка.

Сценарии:

- `2xx` -> экран успеха (`_smiShowSubmitSuccess`).
- `400 + errors` -> ошибки раскладываются по полям (`_smiApplyServerValidationErrors`) и фокус на первое поле.
- `500` -> общий toast «попробуйте позже».
- сеть/таймаут -> отдельный toast по сетевой ошибке.

## 5.6 Маппинг серверных ошибок в поля формы

Сервер возвращает пути в стиле:

- `mainInfo.name: ...`
- `founders[2].email: ...`

На клиенте:

- `founders[2].email` -> `founders.2.email`
- затем корректируется в DOM-путь с учётом merge (`founders` / `founders_phys`),
- если точного пути нет, есть fallback по префиксу.

## 5.7 UI уведомлений

- Используется toast (`_smiShowFormAlert`) внизу экрана.
- `error` остаётся до закрытия.
- `info` скрывается автоматически.

## 6) Как работает `add_forms.js`

Файл отвечает за создание, редактирование, удаление и переиндексацию динамических сущностей:

- учредители физлица,
- учредители юрлица,
- вложенные учредители юрлиц,
- собственники имущества (аналогично иерархии учредителей),
- источники финансирования.

Ключевые техники:

1. **Шаблоны HTML как строки** с параметром пути (`const p = "founders.0"` и т.п.).
2. **Переиндексация после удаления**:
   - пересчитываются `data-path`,
   - обновляются `name` радио-групп.
3. **Снимок/восстановление состояния карточки**:
   - используется `WeakMap savedStates`,
   - сохраняются значения + radio/checkbox состояния,
   - это решает проблему потери данных при переключении edit/save.
4. **Вычисление и контроль суммы долей**:
   - для вложенных блоков используется рекурсивная проверка,
   - результаты отображаются в UI (`data-sum`).
5. **Состояния карточек**:
   - `editing`: можно менять поля,
   - `saved`: показывается компактный preview.

## 7) Как работает `validate_smi_json.php`

`SmiJsonValidator` — единая серверная валидация, независимая от фронта.

Что важно:

- Проверяет структуру и обязательность по шагам.
- Проверяет форматы:
  - email,
  - phone,
  - phoneCode,
  - проценты в диапазонах.
- Проверяет бизнес-правила:
  - либо учредители, либо владелец сетевого издания,
  - ограничения для физлиц («должно быть нет»),
  - суммы долей = 100% (включая рекурсивные вложенные блоки),
  - специальные флаги соответствия требованиям.

Формат ошибок:

- массив строк вида `path: message`,
- пример: `office.phoneCode: корректный код`.

Этот формат нужен клиенту для подсветки конкретных полей.

## 8) Как работает `generate_pdf.php`

Содержит два слоя:

1. **PDF engine (`SmiPdfGenerator`)**
   - принимает payload (string/array),
   - валидирует через `validateSmiJson`,
   - рендерит секции PDF,
   - сохраняет в `uploads/files/form/<timestamp>_smi/`,
   - обычно сохраняет:
     - PDF файл,
     - `input.json`.

2. **HTTP endpoint (конец файла)**
   - `POST`:
     - читает `php://input`,
     - валидирует не пустой JSON,
     - вызывает генератор,
     - возвращает:
       - успех: `{"message":"Заявка успешно принята"}`
       - валидация: `400 {"errors":[...]}`
       - прочее: `500 {"errors":["Ошибка при формировании заявления"]}`
   - локальная отладка на `localhost`:
     - GET может читать `data.js` и генерировать PDF без API-ответа.

## 9) Быстрое добавление нового поля (шпаргалка)

1. Добавьте поле в `1.html` или шаблон `add_forms.js`.
2. Обязательно дайте корректный `data-path`.
3. Если нужно:
   - `required` или `data-required="true"`,
   - `data-validate="..."`,
   - `data-max-percent="..."` для процентов.
4. Если поле зависит от переключателя:
   - отключайте/включайте через `disabled`,
   - у неактивной ветки ставьте `data-ignore-container`.
5. Добавьте серверную проверку в `validate_smi_json.php`.
6. Добавьте использование в рендер PDF (`generate_pdf.php`), если поле должно быть в итоговом документе.

## 10) Диагностика и типовые проблемы

### Проблема: «Поле заполнено, но не уходит в JSON»

Проверьте:

- есть ли `data-path`,
- не находится ли поле внутри `data-ignore-container`,
- не пустое ли значение (пустые строки фильтруются).

### Проблема: «Ошибка сервера не подсвечивает поле»

Проверьте:

- чтобы путь ошибки соответствовал `data-path` после нормализации (`[...]` -> `.n`),
- для динамических массивов — корректность индексов и merge-логики `founders`/`founders_phys`.

### Проблема: «Сложные вложенные учредители/собственники»

Всегда проверяйте:

- переиндексацию после удаления,
- сумму долей на каждом уровне,
- `data-parent-path`/`data-owner-path` в DOM.

## 11) Контракты API, на которые ориентируется фронт

Успех:

```json
{
  "message": "Заявка успешно принята"
}
```

Ошибка валидации:

```json
{
  "errors": [
    "mainInfo.specialization: выберите хотя бы один вариант",
    "office.phoneCode: корректный код"
  ]
}
```

Ошибка сервера:

```json
{
  "errors": [
    "Ошибка при формировании заявления"
  ]
}
```

## 12) Что знать новому разработчику в первую очередь

Если времени мало, сначала изучите:

1. `data-path` + `buildJsonFromForm`.
2. `data-ignore-container` (почему часть полей «исчезает» из валидации/JSON).
3. `add_forms.js` (динамика + reindex + nested structures).
4. Контракт ошибок `validate_smi_json.php` -> `main.js`.
5. `generate_pdf.php` POST endpoint и pipeline генерации.

После этого уже безопасно добавлять поля и бизнес-валидацию без поломки текущей архитектуры.
